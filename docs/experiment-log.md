# RAG 實驗與問題紀錄

本文件保留測試中發現的問題、原因、處理方式與驗證結果。目的是讓調整具有可追溯性，而不是只保存最後看似正確的結果。

## EXP-001：跨分店資料出現在檢索結果

### 現象

使用 `branch_id=taichung` 詢問台北店資訊時，檢索結果出現「台北店菜單總覽」內容。

### 調查

資料庫搜尋條件為：只允許 `branch_id IS NULL` 的全店文件，或與目前選擇分店相同的文件。SQL 條件本身正確。進一步檢查知識文件後發現，台北店 18 道餐點的摘要被放在 `branch: global` 的「菜單製作與出餐標準」中。

### 根因

文件 metadata 與內容範圍不一致。這是 ingestion 前的資料品質問題，不是向量搜尋或 SQL filter 失效。

### 修正

- 從全店共用文件移除台北店菜單總覽。
- 新增 `17-taipei-menu-overview.md`，設定 `branch: taipei`。
- 新增測試，禁止「台北店目前共有」再次出現在 global 文件。

### 驗證標準

- 台中店檢索不得取回「台北店菜單總覽」。
- 台北店詢問餐點總數時，應取回該文件並回答 18 道。

### 驗證結果（2026-07-01）

- 使用 `branch_id=taichung` 詢問「台北店目前共有幾道餐點？」時，top 5 未包含「台北店菜單總覽」。
- 使用 `branch_id=taipei` 詢問相同問題時，「台北店菜單總覽」排名第 1。
- Live Chat 回答「台北店目前共有 18 道正式餐點」，引用「台北店菜單總覽／正式餐點數量」。
- 稽核 `trace_id`：`65a9d140-f937-4f78-ac83-1bb4a0bde3f8`。

## EXP-002：客訴評估文件被錯誤覆蓋

### 現象

評估集期待「客訴與退款處理」，但 `06-customer-complaints.md` 的標題、類型與內容全部是炸爐清潔 SOP。

### 根因

文件編輯時誤將另一份文件內容覆蓋進來。這會造成客訴題的 Recall@5 必然失敗，也產生重複文件標題。

### 修正

- 還原「客訴與退款處理」文件，補上客訴、嚴重過敏、退款與紀錄流程。
- 新增知識庫文件標題唯一性測試。
- 新增評估集預期文件必須存在於知識庫的測試。

### 驗證標準

- `complaint-01` 與 `complaint-02` 能檢索到「客訴與退款處理」。
- 知識庫不存在重複文件標題。

### 驗證結果（2026-07-01）

- 「客人疑似過敏而呼吸困難要怎麼辦？」的 top 1 為「客訴與退款處理／疑似過敏反應」。
- 自動資料品質測試確認文件標題唯一，且評估集引用的來源標題都存在。

## EXP-003：評估預期標題與實際文件不一致

### 現象

`hygiene-01` 期待「員工衛生與洗手規範」，但文件實際標題為「餐廳員工手冊標準」。即使檢索內容正確，評估仍會判定 retrieval miss。

### 修正

將評估案例的 `expected_document` 改為實際文件標題，並由自動測試檢查所有 expected sources 都存在。

## EXP-004：複合問題需要多個來源

### 問題

單一 `expected_document` 無法評估「最後點餐時間＋鮭魚過敏風險」這類需要兩份文件的問題。

### 修正

- `EvaluationCase` 新增 `expected_documents`。
- 複合題只有在所有預期文件都進入 top 5 時，才計為 Recall@5 hit。
- 新增 `compound-01`，要求同時找回台北店打烊 SOP 與全店過敏原指南。

### 已知限制

目前回答策略仍採全答或全拒。若複合問題只有部分證據，整體可能拒答且不保留部分引用。這是保守的產品策略，未來可加入問題拆解與逐項狀態。

### 驗證結果（2026-07-01）

- 複合題 top 5 同時包含「台北店打烊補充 SOP」與「全店菜單與過敏原指南」。
- Live Chat 正確回答 21:30 與鮭魚的花生交叉接觸風險，並分別引用兩份文件。
- 稽核 `trace_id`：`9b473800-1518-4c43-b46e-3a45fbd8c2c5`。

## EXP-005：來源衝突

### 測試情境

同一問題的 Context 同時包含：

- 文件 A：台北店週六最後點餐為 21:30。
- 文件 B：台北店週六最後點餐為 21:00。

### 處理原則

System Prompt 要求來源互相衝突時設定 `abstained=true`。測試確認 ChatService 會保留模型的衝突拒答、清空引用並寫入拒答紀錄。

### 測試邊界

自動測試使用 deterministic fake model，驗證應用層行為，不代表每次 live model 都必然做出相同判斷。正式知識庫不加入刻意衝突文件，避免污染 Demo；live model 的衝突遵循性應在隔離的測試資料庫中另外評估。

## EXP-006：敏感 Context 出現在終端輸出

### 現象

`openai_service.py` 會直接印出完整 Prompt、System Prompt 與模型 Response，`chat_service.py` 也會印出結構化回答與 Token 數。

### 風險

正式環境可能將內部文件、問題或模型結果寫入平台 log，增加資料外洩與 log 成本風險。

### 修正

移除這兩支服務中的直接 `print()`。需要除錯時應改用可設定層級的結構化 logging，並遮蔽敏感內容。CLI 的管理指令仍保留必要的執行結果輸出。

## EXP-007：評估只有摘要，無法定位失敗題

### 現象

`/evaluations/run` 原本只回傳 Recall@5、拒答率、引用有效率與平均延遲。當指標低於 100% 時，無法直接知道失敗案例與實際檢索文件。

### 修正

- 新增逐題 `EvaluationCaseResult`。
- 回傳預期來源、實際檢索文件、引用文件、回答、拒答原因、延遲與各項 pass/fail。
- 新增 `overall_passed`，合併檢索、拒答與引用驗證，但不宣稱它等於語意答案正確率。
- 增加自動測試，確認一般回答題與拒答題的逐題明細。

### 正式評估結果（2026-07-01）

- 案例：31 題
- Recall@5：100%
- 正確拒答率：100%
- Citation validity：100%
- 自動條件通過：31/31
- 平均端到端延遲：4,588 ms
- 失敗案例：0

完整解讀請參考 [`evaluation-report.md`](evaluation-report.md)。

## EXP-008：評估結果沒有持久化

### 現象

`/evaluations/run` 完成後只把 JSON 回傳到終端機。若沒有使用 `curl -o` 手動保存，請求結束後便無法再次查詢，也無法可靠比較不同版本。

### 修正

- 新增 `evaluation_runs` 保存摘要、模型與執行時間。
- 新增 `evaluation_case_results` 保存每題檢索、拒答、引用、回答與延遲。
- 摘要與逐題結果使用同一個 transaction；任一步失敗時全部 rollback。
- `/evaluations/run` 回傳可稽核的 `run_id`。
- 新增 `GET /evaluations/latest` 讀取最近一次完整結果。

### 驗證結果（2026-07-01）

- Migration `004_evaluation_runs` 已套用到 Supabase。
- 3 題 smoke run 成功保存摘要與 3 筆 case results。
- 從資料庫讀回的 latest run 與回傳 `run_id` 完全一致。
- Smoke run：`1e8915cc-12cf-4cd5-995d-01bcd9c1b5d5`。
